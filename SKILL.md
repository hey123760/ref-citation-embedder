# ref-citation-embedder

将参考文献引用标记嵌入 Word DOCX，生成可点击跳转的 REF 域。

## 入口决策流

你的文档是什么状态 → 你应该做什么 → 命令

| 文档状态 | AI 要做的 | 命令 |
|---------|-----------|------|
| 文末有参考文献列表，正文有/无 `[N]` | 分析正文，写好全套 `[N]` 标记 | `--replace` |
| 正文 `[N]` 已存在且完全正确 | 直接跑 | `--replace --keep-numbers` |
| 正文有 `[N]` 但编号冲突/需合并文献列表 | 统一正文 `[N]` 编号，然后跑 | `--replace --keep-numbers` |
| 无参考文献列表也无 `[N]` | 先补参考文献列表，再走场景E | 需用户先补充 |

> **大部分情况走第一行**。AI 的主要工作是保证正文中 `[N]` 正确，脚本只做格式化。

## 最常用模式

### 场景D：正文已有完整手打 [N] → 替换为 REF 域
```bash
python scripts/refcite.py 论文.docx --replace --keep-numbers --verify
```

### 场景E（默认）：AI 分析 → 写全套 [N] 标记 → --replace
AI 分析正文内容，决定各篇文献的引用位置 → 在正文中写入全套 `[N]` 手打标记 → 脚本执行替换。
```bash
python scripts/refcite.py 论文.docx --replace --keep-numbers --verify
```
> 这是**最常用、最推荐**的模式，适用于无 `[N]` 标记或只有部分标记的情况。AI 写完整的 `[N]` 标记后用户可以在 Word 中直接看到并调整。使用 `--keep-numbers` 保持原编号，不加则按首次出现顺序重排。

> ⚠️ **AI 执行规则**：如果文档中有坏域/残域（如 "错误!不能识别的开关参数"），**不要自己写脚本预处理**。直接写 `[N]` 标记后运行 `--replace`，脚本内部的 `clean_refs()` 会自动清理旧域结构。手动预处理（删空 run、删 fldChar、清文本）会导致分页符丢失、句号上标残留等问题。

### 双轨模式：AI/外部工具指定引用位置 → 脚本执行格式化
> 详见 [`docs/DUAL_TRACK.md`](docs/DUAL_TRACK.md) — 外部工具产出映射表，本脚本只执行，互不干扰。

## Quick Start

本 skill 附带可执行脚本，优先使用：

```bash
# 句级映射（推荐）：外部工具指定精确位置
python scripts/refcite.py 论文.docx --smap "19:1:。,24:2:。"

# 自动匹配（兜底）：不提供映射，skill 自行匹配
python scripts/refcite.py 论文.docx --auto

# 替换手打标记（场景D）：正文已有 [N] 标记，原位替换为 REF 域
python scripts/refcite.py 论文.docx --replace

# 替换手打标记 + 保持原编号（不按首次出现顺序重排）
python scripts/refcite.py 论文.docx --replace --keep-numbers
```

## 三种匹配模式

### 模式①：显式引用计划（最可靠）

用户指定哪段引哪篇文献，skill 跳过自动匹配，直接执行格式化。

```
正文P5 → 引用文献 #1
正文P12 → 引用文献 #2
```

> 💡 双轨用法（外部工具产出映射表 + skill 执行）见 [`docs/DUAL_TRACK.md`](docs/DUAL_TRACK.md) — 这是实际最常用的模式：AI 做语义理解决定引哪篇，脚本只做格式化，各司其职。

### 模式②：主题词提示（中等可靠）

用户提供段落主题词，skill 在限定范围内匹配。

```
topic_hints = {"研究背景": [1, 2], "技术方法": [3, 4]}
```

### 模式③：自动关键词匹配（兜底）

无用户输入，skill 自行提取文献标题关键词匹配段落。已知局限：高频词集中匹配。

## 三种使用场景

| 场景 | 说明 | 入口 |
|------|------|------|
| A | 正文 DOCX + 独立参考文献文件 | `--smap` |
| B | 正文有参考文献列表，无引用标记 | `--smap` |
| D | 正文已有手打 `[N]` 标记 | `--replace` 原位替换为 REF 域，默认按首次出现顺序重排编号，加 `--keep-numbers` 保持原编号 |
| E | 正文已有部分 `[N]` 标记，需补充更多引用 | AI 分析后写入新 `[N]` 标记 → `--replace --keep-numbers`（详见 DUAL_TRACK.md 变体） |
| C | 正文无参考文献也无引用标记 | 需用户先补充 |

## 硬性规则

1. **脚本绝不主动修改原文语义内容** — 脚本本身不会删除、移动或改写原文中的任何文字。但在"场景E（AI辅助补充引用）"中，外部 AI 可能在原文中写入新的 `[N]` 手打标记，作为引用计划的一部分。这些标记是格式化工序的输入，不属于脚本对原文语义内容的修改。用户有权在运行脚本前确认或调整这些标记。**简单说：写 `[N]` 是输入准备，不归脚本管。**
2. **rPr 必须每个 run 独立创建** — 禁止共享 rPr（lxml parent 唯一性）
3. **正文已有 `[N]` 标记时禁用启发式匹配** — 直接以现有标记为引用计划
4. **标点归属后半** — REF 插在标点前：`text[1]，next`，非 `text，[1]next`
5. **不要自己写 python-docx 做引用替换** — `refcite.py` 处理了跨 run 匹配、编号重排、书签创建、参考文献重排和编号文字更新。你只需要保证正文的 `[N]` 文本是正确的，其余全部交给 `--replace`。手动操作 XML（拆 run、替换 t.text、处理编号冲突）几乎总是引入新问题，且这些逻辑 refcite.py 内部已经实现。需要修编号 → 修 `[N]` 文本；需要格式化 → 跑 `--replace`。各司其职。

## 注意事项

1. REF 域必须完整：5 run（begin → instrText → separate → 文本 → end），缺一不可
2. 参考文献标题支持「主要参考文献」「【参考文献】」「参考文献」「References」「Bibliography」等多种写法
2. 书签 ID 用 5000+ 大基数，避免与文档既有书签冲突
3. 段落过滤：封面、摘要、关键词、章标题必须跳过
4. 运行前关闭 Word（输出文件被打开会 PermissionError）
5. 备份优先：任何处理前必须先备份原文件，验证不通过则终止处理
6. 统计输出：每次运行打印提取文献数、有效段数、匹配数

## 常见误区

### "错误!不能识别的开关参数。" 或 "错误!未找到引用源。"

这不是 skill 的问题。这是 Word REF 域在**导出 PDF 前没有锁定**导致的。

REF 域是动态代码（按 F9 刷新、Ctrl+F11 锁定）。导出 PDF 时如果域没锁定，Word 会重新计算域，若书签路径失效则显示上述错误。

**正确操作**：在 Word 中按 `Ctrl+A` 全选 → `Ctrl+F11` 锁定域，然后再导出 PDF。锁定后域变为静态文本，PDF 不会报错。

### "引用的[1]变成了 { REF _Ref5001 \h }"

这不是 skill 的问题。这是 Word 进入了**域代码显示模式**（按 `Alt+F9` 切换）。

skill 无法处理这类情况，因为 `[N]` 手打标记在域代码模式下不可见，`--replace` 无法扫描到它们。**需要先在 Word 中按 `Alt+F9` 切换回域结果显示**，再运行 skill。

无论怎么替换 REF 域，导出前不锁定都会出现这个问题。skill 不处理这类情况。
